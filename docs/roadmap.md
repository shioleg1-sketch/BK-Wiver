# Дорожная карта BK-Wiver

## Фаза 0. Подготовка

- утвердить стек `Rust + Tauri + WebRTC`;
- выбрать схему идентификации устройств;
- описать `protobuf` контракты;
- подготовить матрицу платформенных ограничений.

Результат:

- есть архитектурная база;
- понятно, что именно входит в MVP.

## Фаза 1. Control Plane

- `Server` поднимает `HTTP API` и `WebSocket`;
- `Server` отдает web-интерфейс администратора;
- `Desktop` умеет логиниться;
- `Desktop` регистрирует локальное устройство;
- `Desktop` шлет heartbeat;
- `Desktop` показывает список устройств и их онлайн-статус.

Критерий готовности:

- оператор видит удаленную машину в интерфейсе;
- сервер знает, что устройство онлайн;
- администратор может зайти в web-панель и увидеть устройства.

## Фаза 2. Сессия без видео

- создание remote-session;
- выдача временного токена;
- обмен signaling-сообщениями;
- установка `WebRTC` data channel между двумя экземплярами `Desktop`.

Критерий готовности:

- два экземпляра `Desktop` соединяются напрямую или через TURN.

## Фаза 3. Screen View

- захват экрана на удаленном `Desktop`;
- кодирование кадров;
- отображение потока в `Desktop`;
- адаптация качества под канал.

Критерий готовности:

- оператор видит рабочий стол удаленной машины с приемлемой задержкой.

## Фаза 4. Remote Control

- события мыши;
- события клавиатуры;
- clipboard sync при необходимости;
- базовая защита от случайного захвата локального ввода.

Критерий готовности:

- удаленным рабочим столом можно управлять.

## Фаза 5. File Transfer

- просмотр файлов;
- загрузка и скачивание;
- контроль прав и ограничений;
- возобновление передачи при обрывах по возможности.

Критерий готовности:

- можно обмениваться файлами в рамках одной сессии.

## Фаза 6. Hardening

- audit log;
- автообновление;
- crash reporting;
- reverse proxy и TLS для web-панели;
- подпись сборок;
- установщики под платформы.

Критерий готовности:

- продукт можно тестировать на реальных машинах, а не только локально.
