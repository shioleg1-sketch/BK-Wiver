# BK-Wiver

BK-Wiver - заготовка для системы удаленного доступа с единым desktop-приложением и серверной частью.

## Быстрый старт

Минимальный локальный запуск текущего server MVP:

```powershell
docker compose up --build
```

Одинаковый UX на `Windows/Linux/macOS` через `justfile`:

```powershell
just up
```

Если нужны свои порты или пароль БД, создай локальный `.env` по образцу из `.env.example`.

Полезные команды:

- `just bootstrap`
- `just up`
- `just up-detached`
- `just down`
- `just logs`
- `just check`
- `just rebuild`
- `just reset-db`

После старта:

- PostgreSQL доступен на `127.0.0.1:5432`
- Server healthcheck доступен на `http://127.0.0.1:8080/healthz`

Целевые компоненты:

- `Desktop` для установки на `Windows` и `macOS`;
- `Server` для установки на `Ubuntu Server 24.04 x86_64` с web-интерфейсом управления.

Цель проекта:

- удаленный просмотр и управление рабочим столом;
- файловый обмен;
- самохостинг серверной части;
- каталог устройств и пользователей;
- подключение как по прямому пути, так и через relay/TURN.

## Почему не копировать Aspia напрямую

Aspia - хороший референс по разбиению системы на `Host / Client / Console / Router / Relay`, но сам проект распространяется под `GPL-3.0`. Для нового продукта безопаснее использовать его как архитектурный ориентир, а не как источник кода.

Дополнительно важно, что у Aspia:

- `Client` и `Console` есть для `Windows/macOS/Linux`;
- `Router` и `Relay` есть для `Windows/Linux`;
- `Host` доступен только для `Windows`.

Для BK-Wiver это означает, что главная сложность проекта не в UI, а в реализации кроссплатформенного desktop-приложения, которое одновременно умеет быть и клиентом, и принимающей стороной.

## Предлагаемый MVP

Первый реально достижимый MVP:

1. `Desktop` на удаленной машине:
   - регистрация устройства на сервере;
   - стриминг экрана;
   - передача мыши и клавиатуры;
   - базовый обмен файлами.
2. `Desktop` оператора:
   - список устройств;
   - подключение к сессии;
   - окно удаленного рабочего стола;
   - экран статуса разрешений локального устройства;
   - прием входящих подключений.
3. Серверная часть:
   - аутентификация;
   - каталог устройств;
   - сигналинг для установки p2p-сессии;
   - relay/TURN-режим для случаев без прямого соединения;
   - web-интерфейс управления.

## Рекомендуемый стек

Базовый вариант, с которого стоит стартовать:

- `Rust` для `Server` и нативной части `Desktop`;
- `Tauri + TypeScript` для UI desktop-приложения;
- `React/TypeScript` для web-интерфейса администрирования сервера;
- `Protocol Buffers` для контрактов и сообщений;
- `WebRTC` для передачи медиа и data channels;
- `coturn` на первом этапе вместо собственной relay-реализации;
- `SQLite/PostgreSQL` для сервера управления устройствами и учетными данными.

Почему так:

- `Rust` хорошо подходит для фоновых сервисов и нативной интеграции;
- `WebRTC` закрывает NAT traversal, шифрование и транспорт без необходимости сразу писать свой media stack;
- web-интерфейс на сервере избавляет от отдельного админского desktop-приложения;
- собственные `Router/Relay` можно добавить позже, когда MVP подтвердит сценарии.

Следующий практический шаг после этих документов:

1. расширить `proto/` сообщениями для inventory, ввода и файлов;
2. поднять `server` под `Ubuntu Server 24.04 x86_64` как минимальный signaling server;
3. создать `desktop` с heartbeat и регистрацией устройства;
4. добавить в `desktop` список устройств и создание сессии;
5. добавить web-панель управления сервером;
6. добавить в `desktop` прием входящей сессии.
